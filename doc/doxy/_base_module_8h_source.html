37 /**
38  * @brief Base class for all simple modules of a host.
39  *
40  * This method raises
41  * an error if the host state changes to something else than ACTIVE.
42  * Therefore that a sub-classing module can be used in a simulation
43  * where the host state can change it has to override that method
44  * which forces the author to make sure the module reacts well to
45  * host state changes.
46  * Alternatively one can also set a "notAffectedByHostState" parameter
47  * of the module to true.
48  *
49  * The base module additionally provides a function findHost which
50  * returns a pointer to the host module and a function hostIndex to
51  * return the index of the host module. The latter one correspondes to
52  * the index shown in tkenv and comes in very handy for testing and
53  * debugging using tkenv. It is used e.g. in all the 'print' macros
54  * used for debugging.
55  *
56  * There will never be a stand-alone BaseModule module.
57  *
58  * Note: most modules wont derive from BaseModule directly but from
59  * its sub class "BatteryAccess" which extends BaseModule by several
60  * methods for accessing the battery module.
61  *
62  * @see BatteryAccess
63  *
64  * @ingroup baseModules
65  *
66  * @author Steffen Sroka
67  * @author Andreas Koepke
68  */
69 class VEINS_API BaseModule : public cSimpleModule, public cListener {

123  /**
124  * @brief Divide initialization into two stages
125  *
126  * In the first stage (stage==0), modules subscribe to notification.
127  * The first notifications (e.g. about the initial
128  * values of some variables such as RadioState) should take place earliest
129  * in the second stage (stage==1), when everyone interested in them has
130  * already subscribed.
131  * Further one should try to keep calls to other modules out of stage 0 to
132  * assure that the other module had at least once the chance to initialize
133  * itself in stage 0.
